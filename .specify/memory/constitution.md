# Todo Full-Stack Web Application — Hackathon Constitution

## Core Principles

### 1. Spec-Driven Development (Non-Negotiable)
Every feature MUST start as a specification in `/specs/`; Specifications are the source of truth;
Implementation follows specs, not vice versa; Changes require spec updates first, then code
regeneration; No feature exists without a corresponding spec file.

### 2. Agentic Development Workflow (Mandatory)
Write detailed spec → Generate implementation plan → Break into tasks → Implement via Claude Code;
All code generated by Claude Code based on specs; Human role: Write specs, verify outputs, iterate
on specs; Claude Code role: Read specs, generate code, implement features;
Zero tolerance for manual coding or copy-pasting code.

### 3. Architecture Simplicity (Basic Level)
This is a Basic Level hackathon project, not an enterprise system; Simple, working implementation
over complex architecture; No premature optimization; No infrastructure beyond requirements
(no Redis, no load balancers, no Kubernetes); Phase II focus: Working authentication + Task CRUD +
Clean code; Phase III focus: Stateless AI chat + MCP tool calling + DB-persisted conversation state.

### 4. Technology Adherence (Strict)
Only approved technologies from the stack per phase; No substitutions without spec documentation;
No additional frameworks or libraries without justification; Keep dependencies minimal.
Phase II stack is immutable. Phase III stack additions are additive only — they do not replace
Phase II components.

### 5. Security-First Approach (Mandatory)
All authentication MUST use JWT with HS256 algorithm; Passwords MUST be hashed with bcrypt;
HTTP-only cookies for refresh tokens; User data isolation required; SQL injection prevention via
SQLModel ORM; XSS prevention via proper cookie settings.
**CONFIRMED DECISION**: Better Auth is NOT used. The existing JWT/bcrypt system is the only
authentication method across all phases. This decision is locked and must not be reversed.

### 6. Parallel Development Coordination
Backend and frontend teams work in parallel with synchronized integration points;
Shared API contracts and data models; Regular integration testing between services;
Consistent error handling across all layers.

### 7. MCP Server Standards (Phase III — Mandatory)
MCP server MUST expose exactly 5 tools: add_task, list_tasks, complete_task, delete_task,
update_task; Each tool MUST accept user_id to enforce data isolation; MCP tools MUST be stateless
— all state persisted to database; MCP server runs as stdio subprocess invoked by the Agents SDK
(no separate HTTP deployment required for hackathon); Tool descriptions MUST be explicit enough
for the AI agent to select the correct tool without ambiguity.

### 8. AI Agent Behavior Standards (Phase III — Mandatory)
Agent MUST confirm every action with a friendly natural language response; Agent MUST handle
task-not-found errors gracefully without crashing; Agent MUST use list_tasks before delete_task
when user references a task by name (not ID); Conversation history MUST be loaded from DB on
every request (stateless server, stateful DB); Agent MUST NOT fabricate task IDs — always derive
from tool results.

## Key Standards

### Technology Stack (Immutable)

**Phase II (Preserved — do not modify):**
- **Frontend**: Next.js 16+ with App Router, TypeScript, Tailwind CSS only
- **Backend**: Python FastAPI, SQLModel ORM only
- **Database**: Neon Serverless PostgreSQL only (no Redis, no MongoDB, no SQLite)
- **Authentication**: HTTP-only cookies + JWT with HS256 — NO Better Auth, NO OAuth, NO sessions
- **Containerization**: Docker Compose for local development only

**Phase III (Additive — do not remove Phase II entries):**
- **Chat UI**: OpenAI ChatKit (`@openai/chatkit`) — React component for streaming chat interface
- **AI Framework**: OpenAI Agents SDK (Python `openai-agents` package)
- **LLM Provider**: Groq (free tier) — `llama-3.3-70b-versatile` model via OpenAI-compatible API
- **MCP Server**: Official MCP SDK (Python `mcp` package) — stdio transport
- **Streaming**: Server-Sent Events (SSE) — permitted ONLY for `/api/{user_id}/chat` endpoint

### Authentication Architecture (Fixed — All Phases)
Algorithm: HS256 (symmetric) with AUTH_SECRET environment variable;
Access tokens: 15 minutes lifetime, stored in-memory on frontend;
Refresh tokens: 7 days lifetime, stored in HTTP-only cookies + PostgreSQL;
Cookie settings: HttpOnly=true, Secure=true (production), SameSite='lax';
No token rotation, no blacklisting, no refresh token hashing (keep it simple);
Backend validates JWT on every request, extracts user_id, filters data by user.
**Phase III chat endpoint uses the same get_current_user() dependency — no new auth logic.**

### Database Schema Standards
Primary keys: Auto-increment integers (not UUIDs, not strings) for ALL tables;
Foreign keys: MUST match parent table type (integer → integer);
User ID consistency: ALL user_id fields MUST be integers across all tables;
Timestamps: created_at, updated_at where applicable;
Indexes: On foreign keys and frequently queried fields;
Constraints: NOT NULL, UNIQUE, DEFAULT values specified explicitly.

**Phase II tables (existing):**
- `users`: id (int PK), email, username, hashed_password, is_active, created_at
- `tasks`: id (int PK), user_id (int FK→users.id), title, description, completed, created_at, updated_at
- `refreshtoken`: id (int PK), user_id (int FK→users.id), token, expires_at, revoked, created_at

**Phase III tables (additive):**
- `conversations`: id (int PK), user_id (int FK→users.id), created_at, updated_at
- `messages`: id (int PK), user_id (int FK→users.id), conversation_id (int FK→conversations.id),
  role (TEXT: 'user'|'assistant'), content (TEXT), created_at

### API Conventions (Strict)
All routes under `/api/` prefix; RESTful patterns for CRUD; User ID in URL for user-specific
resources; JWT required in Authorization header: `Bearer {token}`; Responses: JSON only
(except SSE stream); Status codes: 200, 201, 401, 404, 500.
**SSE permitted exclusively for `POST /api/{user_id}/chat`** — no other endpoints may use SSE.
No GraphQL, no WebSockets.

**Phase III Chat Endpoint:**
- `POST /api/{user_id}/chat` — Send message, receive streaming AI response
- Request: `{ conversation_id?: int, message: string }`
- Response: `{ conversation_id: int, response: string, tool_calls: array }`

### MCP Tools Specification (Phase III — Fixed Contract)

| Tool | Parameters | Returns |
|------|-----------|---------|
| `add_task` | user_id (str, req), title (str, req), description (str, opt) | task_id, status, title |
| `list_tasks` | user_id (str, req), status (str, opt: all/pending/completed) | array of task objects |
| `complete_task` | user_id (str, req), task_id (int, req) | task_id, status, title |
| `delete_task` | user_id (str, req), task_id (int, req) | task_id, status, title |
| `update_task` | user_id (str, req), task_id (int, req), title (str, opt), description (str, opt) | task_id, status, title |

### Conversation Flow — Stateless Request Cycle (Phase III)
1. Receive user message via POST /api/{user_id}/chat
2. Validate JWT via get_current_user() (existing dependency)
3. Fetch conversation history from `messages` table
4. Store user message in `messages` table
5. Run OpenAI Agent with MCPServerStdio — agent calls MCP tools as needed
6. Store assistant response in `messages` table
7. Stream response back to ChatKit via SSE
8. Server holds NO in-memory state between requests

### Security Requirements (Mandatory — All Phases)
Secrets in environment variables only (AUTH_SECRET, DATABASE_URL, GROQ_API_KEY);
Passwords hashed with bcrypt; SQL injection prevention via SQLModel ORM;
XSS prevention via HTTP-only cookies; CSRF protection via SameSite cookies;
Input validation on all endpoints; User data isolation enforced at every layer.
**Phase III addition**: MCP tools MUST validate user_id against authenticated user — agent-supplied
user_id MUST match the JWT-authenticated user_id. Tool calls with mismatched user_id MUST be rejected.

### Frontend Patterns (Required)
Server components by default; Client components only for interactivity ('use client' directive);
All API calls through `/lib/api.ts` (no direct fetch in components — applies to chat calls too);
Tailwind CSS for styling; TypeScript strict mode; Error boundaries for error handling.

### Backend Patterns (Required)
FastAPI dependency injection for database sessions; Pydantic models for request/response;
HTTPException for error handling; SQLModel for all DB operations; Environment variables via
pydantic-settings; CORS configured for frontend origin only.

## Constraints

### Absolute Prohibitions (Will Fail Review)
❌ Manual coding or editing generated code
❌ Copy-pasting code from external sources
❌ Technologies not in approved stack (Redis, Alembic, Better Auth, etc.)
❌ RS256 or asymmetric JWT encryption
❌ Storing tokens in localStorage
❌ User IDs as strings in database
❌ Hashing refresh tokens in database
❌ Raw SQL queries with user input
❌ Inline styles or CSS-in-JS
❌ API calls outside `/lib/api.ts`
❌ Business logic in frontend components
❌ Authentication logic mixed with business logic
❌ SSE on any endpoint other than `/api/{user_id}/chat`
❌ MCP tools that bypass user_id validation
❌ Storing AI API keys (GROQ_API_KEY) in frontend code or committed files
❌ Better Auth or any alternative auth library — existing JWT/bcrypt is final

### Development Constraints
All features MUST have corresponding spec file; Spec MUST be read before implementation;
Changes require spec update → regenerate code; No feature creep beyond Basic Level requirements;
No premature optimization; Database migrations documented in specs first.

### File Naming Constraints
Specs: Lowercase with hyphens; Components: PascalCase (.tsx); API files: Lowercase (.ts, .py);
No spaces in filenames.

### Deployment Constraints
Local development with Docker Compose; Simple production deployment;
Frontend on Vercel (taskify-ainative.vercel.app); Backend on Render (Render free tier);
MCP server runs as subprocess within Render backend — no separate service;
No load balancers, no replicas, no Kubernetes.

### When to Update Specs
