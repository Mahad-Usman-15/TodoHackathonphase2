
Project: Todo Full-Stack Web Application - Hackathon Phase II

## Core Principles

### 1. Spec-Driven Development (Non-Negotiable)
- Every feature MUST start as a specification in `/specs/`
- Specifications are the source of truth
- Implementation follows specs, not vice versa
- Changes require spec updates first, then code regeneration
- No feature exists without a corresponding spec file

### 2. Agentic Development Workflow (Mandatory)
- Write detailed spec → Generate implementation plan → Break into tasks → Implement via Claude Code
- All code generated by Claude Code based on specs
- Human role: Write specs, verify outputs, iterate on specs
- Claude Code role: Read specs, generate code, implement features
- Zero tolerance for manual coding or copy-pasting code

### 3. Architecture Simplicity (Basic Level)
- This is a Basic Level hackathon project, not an enterprise system
- Simple, working implementation over complex architecture
- No premature optimization
- No infrastructure beyond requirements (no Redis, no load balancers, no Kubernetes)
- Focus: Working authentication + Task CRUD + Clean code

### 4. Technology Adherence (Strict)
- Only approved technologies from the stack
- No substitutions without spec documentation
- No additional frameworks or libraries without justification
- Keep dependencies minimal

## Key Standards

### Technology Stack (Immutable)
- **Frontend**: Next.js 16+ with App Router, TypeScript, Tailwind CSS only
- **Backend**: Python FastAPI, SQLModel ORM only
- **Database**: Neon Serverless PostgreSQL only (no Redis, no MongoDB, no SQLite)
- **Authentication**: HTTP-only cookies + JWT with HS256 (not RS256, not OAuth, not sessions)
- **Containerization**: Docker Compose for local development only

### Authentication Architecture (Fixed)
- Algorithm: HS256 (symmetric) with SECRET_KEY environment variable
- Access tokens: 15 minutes lifetime, stored in-memory on frontend
- Refresh tokens: 7 days lifetime, stored in HTTP-only cookies + PostgreSQL
- Cookie settings: HttpOnly=true, Secure=true (production), SameSite='lax'
- No token rotation, no blacklisting, no refresh token hashing (keep it simple)
- Backend validates JWT on every request, extracts user_id, filters data by user

### Database Schema Standards
- Primary keys: Auto-increment integers (not UUIDs, not strings)
- Foreign keys: Match parent table type (integer → integer)
- Timestamps: created_at, updated_at where applicable
- Indexes: On foreign keys and frequently queried fields
- Constraints: NOT NULL, UNIQUE, DEFAULT values specified explicitly
- Table names: Lowercase, plural (users, tasks, refreshtoken)

### API Conventions (Strict)
- All routes under `/api/` prefix
- RESTful patterns: GET /api/tasks, POST /api/tasks/{id}, etc.
- User ID in URL: `/api/{user_id}/tasks` for user-specific resources
- JWT required in Authorization header: `Bearer {token}`
- Responses: JSON only, consistent error format
- Status codes: 200 (success), 201 (created), 401 (unauthorized), 404 (not found), 500 (server error)
- No GraphQL, no WebSockets, no SSE

### Security Requirements (Mandatory)
- Secrets in environment variables only (SECRET_KEY, DATABASE_URL)
- Passwords hashed with bcrypt (never plain text)
- SQL injection prevention via SQLModel ORM (no raw SQL from user input)
- XSS prevention via HTTP-only cookies (no localStorage for tokens)
- CSRF protection via SameSite cookies
- Input validation on all endpoints
- User data isolation (user can only access their own tasks)

### Frontend Patterns (Required)
- Server components by default
- Client components only for interactivity ('use client' directive)
- All API calls through `/lib/api.ts` (no direct fetch in components)
- Tailwind CSS for styling (no inline styles, no CSS modules)
- TypeScript strict mode enabled
- Error boundaries for error handling

### Backend Patterns (Required)
- FastAPI dependency injection for database sessions
- Pydantic models for request/response validation
- HTTPException for error handling (no generic Exception)
- SQLModel for all database operations (no raw SQL)
- Environment variables via pydantic-settings
- CORS configured for frontend origin only

  Backend Architecture Layers

  1. Middleware Layer

  - Auth Middleware: Validates JWT tokens using AUTH_SECRET
  - CORS Middleware: Configures cross-origin policies
  - Logging Middleware: Tracks requests for debugging
  - Rate Limiting: Prevents abuse
  - Error Handling: Centralized exception management

  2. Route Layer (Controllers)

  - Auth Routes: /api/auth/* (login, register, refresh)
  - User Routes: /api/users/* (profile management)
  - Task Routes: /api/{user_id}/tasks/* (CRUD operations)
  - Validation: Pydantic models for request/response validation
  - Dependency Injection: Services injected into route handlers

  3. Service Layer (Business Logic)

  - User Service: Registration, authentication, password management
  - Task Service: Task creation, filtering, ownership validation
  - Token Service: JWT creation, refresh, validation
  - Security Service: Password hashing, token validation
  - Validation Service: Business rule enforcement

  4. Model Layer (Data Layer)

  - SQLModel Classes: User, Task, RefreshToken models
  - Database Sessions: Transaction management
  - Query Builders: Type-safe database queries
  - Relationship Managers: Foreign key handling

  5. Database Layer

  - Neon Serverless PostgreSQL: Auto-scaling database
  - Connection Pooling: Managed by SQLModel
  - Indexes: Optimized for user_id and task status queries

  Authentication Flow

  1. Login Flow

  1. User enters credentials → Frontend Form
  2. Frontend sends credentials to backend → POST /api/auth/login
  3. Backend validates credentials against database
  4. Backend creates JWT access token (15 min expiry)
  5. Backend creates refresh token in database
  6. Backend sets refresh token in HttpOnly cookie
  7. Backend returns access token in response
  8. Frontend stores access token in-memory
  9. Frontend redirects to dashboard

  2. API Request Flow

  1. Frontend makes API request
  2. Auth middleware extracts access token from headers
  3. JWT validated using AUTH_SECRET
  4. User ID extracted from token claims
  5. Request processed with user context
  6. Database queries filtered by user_id

  3. Token Refresh Flow

  1. Access token expires (401 response)
  2. Frontend detects expired token
  3. Frontend sends refresh token (cookie) to backend
  4. Backend validates refresh token against database
  5. Backend creates new access token
  6. Frontend updates in-memory access token
  7. Original request retried

  Phase Integration Strategy

  Phase 1: Foundation (Always Integrated)

  - Basic auth endpoints + simple login page
  - Empty dashboard showing user context
  - Database connection established
  - Minimal working flow: login → dashboard

  Phase 2: Core Features (Iterative Enhancement)

  - Task CRUD endpoints + task list UI
  - Real-time data fetching
  - Form validation and error handling
  - Each feature incrementally adds to working system

  Phase 3: Advanced Features (Layered Addition)

  - Filtering, sorting, search
  - Advanced UI components
  - Performance optimizations
  - Each addition maintains full system functionality

  Integration Maintenance Strategies:

  1. Feature Flags: Enable/disable features without breaking main flow
  2. Mock Data: Fallback data when backend services unavailable
  3. Progressive Enhancement: Core functionality always works
  4. Contract Testing: Validate API contracts continuously
  5. Automated Testing: End-to-end tests for critical flows

  Potential Failure Points & Mitigation

  1. Authentication Failures

  - Risk: Session invalidation, token expiration
  - Mitigation: Automatic refresh, graceful degradation, clear error states

  2. Database Connectivity

  - Risk: Connection timeouts, query failures
  - Mitigation: Connection pooling, retry logic, circuit breakers

  3. Frontend-Backend Drift

  - Risk: API contract changes, type mismatches
  - Mitigation: Shared type definitions, contract testing, versioned APIs

  4. Security Vulnerabilities

  - Risk: JWT manipulation, SQL injection, XSS
  - Mitigation: Input validation, parameterized queries, security headers

  5. Performance Bottlenecks

  - Risk: Slow queries, large payloads, memory leaks
  - Mitigation: Database indexing, pagination, lazy loading

  6. State Management Complexity

  - Risk: Inconsistent states, race conditions
  - Mitigation: Centralized state, optimistic updates, conflict resolution

  Design Rationale

  Why This Architecture Prevents Late-Stage Issues:

  1. Integrated from Day One: No backend-only or frontend-only phases mean constant integration testing
  2. Contract-First Approach: API specifications prevent frontend-backend misalignment
  3. Layered Separation: Clear boundaries allow independent development and testing
  4. Security Built-In: Authentication and authorization from foundation prevents retrofitting issues
  5. Spec-Driven Development: Architecture supports specification-first workflow
  6. Environment Consistency: Same patterns work across dev, test, prod
  7. Observability: Logging and monitoring built into middleware layer
  8. Scalability Ready: Service layer abstraction allows for microservices transition

  This architecture ensures that the system remains fully functional at every development stage while maintaining security, performance, and maintainability throughout
  the hackathon timeline.
